# RF Trace Report — Large Trace Compact Serialization: Analysis & Requirement Notes

## Context

This document captures the full analysis, reasoning, and design decisions behind **Requirement 35: Large Trace Compact Serialization**, added to the spec after profiling the pipeline against a 610,051-span fixture.

## Root Cause Analysis: Why Is the HTML 153 MB?

The embedded JSON data accounts for **152.6 MB** of the 152.8 MB HTML file.
JS viewer code is only 0.157 MB. CSS is 0.030 MB.

### Field-level breakdown (610,051 nodes)

| Category | Size | % of total |
|---|---|---|
| JSON key name repetition | 77.5 MB | 51% |
| Empty/default values | 28.3 MB | 19% |
| Timing data (start/end/elapsed) | 26.7 MB | 18% |
| Actual useful data (names, IDs, args, statuses) | 20.1 MB | 13% |

### Unique value analysis (the repetition problem)

| Category | Unique values | Total occurrences |
|---|---|---|
| Keyword names | 10 | ~600,000 |
| Keyword types | 1 (`"KEYWORD"`) | 600,000 |
| Statuses | 3 (`PASS`, `FAIL`, `SKIP`) | 610,051 |
| Sources | 51 | ~600,000 |

Only 10 unique keyword names across 600,000 keyword nodes. The same string `"Log"` is serialized 60,000 times. This is the core inefficiency.

## Optimization Strategies (Ranked by Impact)

### Strategy 1: Omit empty defaults (saves ~28 MB, zero complexity)

Stop serializing fields at their default values:
- `"doc":""` → omit
- `"status_message":""` → omit
- `"events":[]` → omit
- `"children":[]` → omit
- `"tags":[]` → omit
- `"lineno":0` → omit

The JS viewer already uses `if (data.doc)` style checks — `undefined` is falsy just like `""`.
No JS changes needed. Pure Python change in `generator.py`.

**Estimated savings: ~28 MB (19% reduction)**

### Strategy 2: Short key names (saves ~44 MB, low complexity)

Replace verbose JSON keys with 1-2 character aliases:

| Original key | Short alias |
|---|---|
| `"keyword_type"` | `"kt"` |
| `"status_message"` | `"sm"` |
| `"elapsed_time"` | `"el"` |
| `"start_time"` | `"st"` |
| `"end_time"` | `"et"` |
| `"children"` | `"ch"` |
| `"events"` | `"ev"` |
| `"attributes"` | `"at"` |
| `"status"` | `"s"` |
| `"name"` | `"n"` |
| `"type"` | `"t"` |
| `"doc"` | `"d"` |
| `"lineno"` | `"ln"` |
| `"args"` | `"a"` |
| `"tags"` | `"tg"` |
| `"metadata"` | `"md"` |
| `"keywords"` | `"kw"` |

Embed the key-mapping table as `"km"` in the wrapper JSON object. The JS viewer decodes it on load.
Add a format version field `"v": 1` for future compatibility.

**Estimated savings: ~44 MB (29% reduction)**

### Strategy 3: String intern table (saves ~14 MB, medium complexity)

Collect all unique string values that appear more than once across the serialized data.
Store them in an array (`"it"` in the wrapper). Replace each repeated occurrence with its integer index.

Example: `"Log"` appears 60,000 times. Store `"Log"` once at index 0. Replace all occurrences with `0`.
`"KEYWORD"` appears 600,000 times. Store once at index 1. Replace all with `1`.

**Estimated savings: ~14 MB (9% reduction)**

### Strategy 4: Gzip + base64 embedding (saves ~145 MB, medium complexity)

The embedded JSON compresses 95% with gzip (152 MB → 7.8 MB).

Embed as a base64-encoded gzip blob:
```html
<script>window.__RF_TRACE_DATA_GZ__ = "H4sIAAAAA...";</script>
```

The JS viewer decompresses at load time using the browser's native `DecompressionStream` API
(available in Chrome 80+, Firefox 113+, Safari 16.4+ — all modern browsers since 2023).

```javascript
async function decompressData(b64) {
  const bytes = Uint8Array.from(atob(b64), c => c.charCodeAt(0));
  const ds = new DecompressionStream('gzip');
  const writer = ds.writable.getWriter();
  writer.write(bytes);
  writer.close();
  const chunks = [];
  for await (const chunk of ds.readable) chunks.push(chunk);
  return JSON.parse(new TextDecoder().decode(
    new Uint8Array(chunks.reduce((a, b) => [...a, ...b], []))
  ));
}
```

**Estimated savings: ~145 MB (95% reduction). HTML goes from 153 MB to ~11 MB.**

### Strategy 5: CLI filtering flags (new feature, reduces data before serialization)

For users who only care about failures or want to limit report size:

- `--max-keyword-depth N` — truncate keyword nesting beyond depth N. Mark truncated nodes with `truncated: true` so the JS viewer shows "… N keywords hidden".
- `--exclude-passing-keywords` — omit PASS keyword spans entirely. Suite and test spans always retained. Useful when you only care about what failed.
- `--max-spans N` — hard cap on total spans. Prioritize FAIL > SKIP > PASS, shallowest first. Emit warning to stderr: "Warning: trace truncated to N spans (M spans omitted)".

These are pre-serialization filters — they reduce the data model before it ever gets serialized.

### Combined impact estimate

| Strategy | Savings | HTML size after |
|---|---|---|
| Baseline | — | 153 MB |
| + Omit defaults | 28 MB | 125 MB |
| + Short keys | 44 MB | 81 MB |
| + String intern | 14 MB | 67 MB |
| + Gzip embed | 95% of 67 MB | **~3-4 MB** |

Strategies 1+2+3 alone cut the HTML from 153 MB to ~67 MB with no loss of functionality.
Adding strategy 4 (gzip on top of the already-compacted data) gets it to ~3-4 MB.

## Requirement 35 (as added to requirements.md)

### Requirement 35: Large Trace Compact Serialization

**User Story:** As a test engineer working with very large test suites (500,000+ spans), I want the generated HTML report to be as small as possible so that it loads quickly in the browser and can be stored or shared without excessive disk usage.

#### Acceptance Criteria

1. WHEN the Report_Generator serializes trace data for embedding, THE Report_Generator SHALL omit fields that are at their default empty values (`""`, `[]`, `{}`, `0`) from the JSON output, reducing payload size without losing information (the JS viewer SHALL treat missing fields as their defaults).

2. THE Report_Generator SHALL use a compact key-mapping table to replace verbose JSON field names with short aliases (e.g., `"keyword_type"` → `"kt"`, `"status_message"` → `"sm"`, `"start_time"` → `"st"`, `"end_time"` → `"en"`, `"elapsed_time"` → `"el"`, `"children"` → `"ch"`, `"events"` → `"ev"`, `"attributes"` → `"at"`, `"status"` → `"s"`, `"name"` → `"n"`, `"type"` → `"t"`, `"doc"` → `"d"`, `"lineno"` → `"ln"`, `"args"` → `"a"`, `"tags"` → `"tg"`, `"metadata"` → `"md"`) in the embedded JSON, with the key-mapping table itself embedded in the HTML so the JS viewer can decode it.

3. THE Report_Generator SHALL build a string lookup table (intern table) for repeated string values: collect all unique string values that appear more than once across the serialized data, store them in an array, and replace each repeated occurrence with its integer index into that array. The JS viewer SHALL decode the intern table on load.

4. WHEN the CLI is invoked with `--compact-html` flag, THE Report_Generator SHALL apply all compact serialization optimizations (omit defaults, short keys, string intern table) to the embedded JSON.

5. WHEN the CLI is invoked with `--gzip-embed` flag, THE Report_Generator SHALL gzip-compress the embedded JSON data, base64-encode it, and embed it as a string constant. The JS viewer SHALL decompress it at load time using the browser's native `DecompressionStream` API (available in all modern browsers since 2023).

6. WHEN the CLI is invoked with `--max-keyword-depth N`, THE Report_Generator SHALL truncate the span tree at keyword nesting depth N, omitting deeper keyword children from the embedded data. A visual indicator SHALL be shown in the JS viewer for truncated nodes.

7. WHEN the CLI is invoked with `--exclude-passing-keywords`, THE Report_Generator SHALL omit keyword spans with PASS status from the embedded data, retaining only FAIL, SKIP, and NOT_RUN keyword spans (suite and test spans are always retained regardless of status).

8. WHEN the CLI is invoked with `--max-spans N`, THE Report_Generator SHALL limit the total number of spans embedded in the HTML to N, prioritizing FAIL spans, then SKIP spans, then PASS spans in descending order of depth (shallowest first). A warning SHALL be emitted to stderr indicating how many spans were omitted.

9. THE JS_Viewer SHALL detect and decode compact serialization format automatically on load: check for the presence of the key-mapping table and intern table in the embedded data, and transparently expand the data to the full format before rendering.

10. THE compact serialization format SHALL be versioned (a `"v"` field in the wrapper object) so that future format changes can be detected and handled by the JS viewer.

11. WHEN both `--compact-html` and `--gzip-embed` flags are provided together, THE Report_Generator SHALL first apply compact serialization (omit defaults, short keys, intern table) and then gzip-compress the resulting compact JSON, so both optimizations compose and the maximum size reduction is achieved.

## Design Notes: JS Compact Decoder

The decoder in `app.js` needs to handle two transformations:

**1. Expand short keys back to full names:**
```javascript
const KEY_MAP = {
  n: 'name', s: 'status', d: 'doc', a: 'args', ln: 'lineno',
  kt: 'keyword_type', sm: 'status_message', st: 'start_time',
  et: 'end_time', el: 'elapsed_time', ch: 'children', ev: 'events',
  at: 'attributes', t: 'type', tg: 'tags', md: 'metadata', kw: 'keywords'
};

function expandNode(node, km, it) {
  const expanded = {};
  for (const [k, v] of Object.entries(node)) {
    const fullKey = km[k] || k;
    expanded[fullKey] = expandValue(v, km, it);
  }
  return expanded;
}

function expandValue(v, km, it) {
  if (typeof v === 'number' && it && v < it.length) return it[v]; // intern index
  if (Array.isArray(v)) return v.map(item =>
    typeof item === 'object' && item !== null ? expandNode(item, km, it) : expandValue(item, km, it)
  );
  if (typeof v === 'object' && v !== null) return expandNode(v, km, it);
  return v;
}
```

**2. Wrapper JSON structure:**
```json
{
  "v": 1,
  "km": {"n": "name", "s": "status", ...},
  "it": ["Log", "KEYWORD", "PASS", "Should Be Equal", ...],
  "data": { ...compressed span tree... }
}
```

**3. Backward compatibility check:**
```javascript
function decodeTraceData(raw) {
  if (!raw.v) return raw; // old format, pass through
  const { km, it, data } = raw;
  return expandNode(data, km, it);
}
```

## Design Notes: Gzip Embed HTML Template

When `--gzip-embed` is active, the HTML template changes from:
```html
<script id="trace-data" type="application/json">{json_data}</script>
```

To:
```html
<script>window.__RF_TRACE_DATA_GZ__ = "{base64_gzip_data}";</script>
```

And `app.js` initialization changes from synchronous:
```javascript
const raw = JSON.parse(document.getElementById('trace-data').textContent);
init(decodeTraceData(raw));
```

To async:
```javascript
async function loadTraceData() {
  if (window.__RF_TRACE_DATA_GZ__) {
    const bytes = Uint8Array.from(atob(window.__RF_TRACE_DATA_GZ__), c => c.charCodeAt(0));
    const ds = new DecompressionStream('gzip');
    const writer = ds.writable.getWriter();
    writer.write(bytes); writer.close();
    const chunks = [];
    for await (const chunk of ds.readable) chunks.push(chunk);
    const json = new TextDecoder().decode(new Uint8Array(chunks.flatMap(c => [...c])));
    return decodeTraceData(JSON.parse(json));
  }
  return decodeTraceData(JSON.parse(document.getElementById('trace-data').textContent));
}
loadTraceData().then(data => init(data));
```

## Correctness Properties for Testing

### Property 27: Compact serialization round-trip
For any span tree, serializing with compact format (omit defaults + short keys + intern table) then decoding with the JS decoder logic (ported to Python for testing) produces data equivalent to the original full-format serialization.

### Property 28: Gzip embed round-trip
For any JSON string, gzip-compressing at level 9 then base64-encoding then base64-decoding then gzip-decompressing produces the original JSON bytes.

### Property 29: Span truncation correctness
For any span tree and any N, `_limit_spans(tree, N)` produces at most N spans total, and for any FAIL span in the original tree that is within the top N by priority, it appears in the truncated output.
